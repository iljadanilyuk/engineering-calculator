# Public Conversion, Example Delivery, And Questionnaire Roadmap

Дата: 2026-07-17  
Проект: инженерный калькулятор ИП Позняк  
Рабочая папка: `E:\vc\poznyak-engineering-calculator`

## Контекст

После production launch и русификации админки пользователь уточнил важные продуктовые изменения для публичной воронки.

Текущая проблема: публичная страница может отдавать слишком много ценности без контакта. Если посетитель просто нажимает кнопку и открывает пример проекта, он может уйти, а у владельца не остается способа продолжить взаимодействие.

Нужно перестроить публичную воронку так, чтобы:

- примеры проектов не отдавались анонимно;
- предварительное коммерческое предложение выдавалось после контакта;
- у пользователя был выбор между быстрым предварительным КП и подробным опросным листом;
- подробный опросник собирал данные для технического задания пошагово;
- Telegram можно было использовать как канал доставки примеров/КП и в будущем как источник уточнений по проекту.

## Продуктовая логика

### 1. Примеры проектов

Сейчас примеры проектов открываются как обычные PDF-ссылки. Это хорошо для доверия, но плохо для сбора контактов.

Новая логика:

1. Пользователь нажимает `Получить пример проекта`.
2. Система просит имя, телефон и согласие.
3. После отправки контакта пользователь получает пример:
   - либо сразу ссылкой на странице;
   - либо через Telegram-бота;
   - либо другим согласованным способом.
4. В админке создается/обновляется лид с источником `example_request`.

Важная развилка:

- Если цель только не потерять контакт, достаточно скрыть прямые кнопки и выдавать ссылку после формы.
- Если цель действительно не дать открыть PDF без контакта, файлы нельзя держать как открытые static files в `website/public/project-examples`. Тогда нужно перенести выдачу на backend token-protected routes или другое закрытое хранилище.

Рекомендация для v1:

- Сделать lead-gated выдачу через форму как основной путь.
- Сразу заложить backend token route для примеров, чтобы PDF-ссылки были непредсказуемыми и могли логироваться.
- Telegram-доставку сделать опциональным каналом, а не единственным путем, потому что не каждый клиент захочет уходить в Telegram до просмотра примера.

### 2. Получение КП после расчета

Сейчас пользователь выбирает площадь и разделы, нажимает `Получить КП`, оставляет имя/телефон и получает КП/PDF.

Новая логика:

1. Пользователь выбирает площадь и разделы проекта в калькуляторе.
2. Нажимает `Получить коммерческое предложение`.
3. Попадает на новую страницу выбора:
   - `Пройти подробный опросник` - для более точного КП и будущего технического задания;
   - `Скачать предварительное КП по расчету` - по тем данным, которые уже введены в калькуляторе.
4. Если пользователь выбирает предварительное КП:
   - появляется форма имя/телефон/согласие;
   - после отправки создается лид;
   - backend пересчитывает сумму;
   - генерируется preliminary proposal/PDF;
   - пользователь получает ссылку/документ, возможно через Telegram.
5. Если пользователь выбирает подробный опросник:
   - стартует пошаговый wizard;
   - ответы сохраняются;
   - прогресс виден пользователю;
   - итог привязывается к лиду/карточке проекта;
   - после контакта можно выдать более точное КП или создать задачу для менеджера.

### 3. Подробный опросный лист / техническое задание

Источник: `docs/design/Опросный лист.xlsx`.

Опросник должен быть не длинной формой на одной странице, а последовательным сценарием:

- вопросы идут по разделам;
- пользователь видит прогресс;
- на каждый вопрос есть варианты ответов;
- можно выбрать `Пока не знаю`;
- можно выбрать `Пропустить`;
- можно ввести свой вариант;
- ответы сохраняются по ходу прохождения;
- можно продолжить позже по ссылке;
- в админке ответы отображаются как черновик технического задания.

Важно: не нужно заставлять пользователя проходить весь опросник до первого контакта, если это снижает конверсию. Лучше дать понятную вилку:

- быстрый путь: предварительное КП по данным калькулятора;
- точный путь: подробный опросник для уточнения объекта.

### 4. Telegram как канал

Telegram может выполнять несколько ролей:

1. Доставка примера проекта после контакта.
2. Доставка предварительного КП/PDF.
3. Канал продолжения диалога после заявки.
4. В будущем - проектная группа, из которой агент ежедневно извлекает факты и обновляет черновик ТЗ в админке.

Рекомендация:

- Не делать Telegram единственным путем получения документов.
- Сначала реализовать веб-форму и token links.
- Затем добавить Telegram delivery как дополнительный канал:
  - кнопка `Получить в Telegram`;
  - bot deep link;
  - привязка Telegram chat/user к lead/project;
  - лог доставки в админке.

## Новые задачи

### PZK-017 - Lead-Gated Project Examples And Preliminary Offer Delivery

Смысл: перестать отдавать examples/PDF без контакта.

Основные изменения:

- заменить прямые public example PDF buttons на CTA с lead capture;
- создать backend endpoint для example request;
- сохранять имя/телефон/consent/source;
- выдавать example link только после контакта;
- подумать, нужно ли убрать PDF из полностью публичной static-директории;
- добавить admin visibility: источник лида `Запрос примера проекта`;
- сохранить текущие PDF examples как proof assets, но изменить механику доступа.

### PZK-018 - Commercial Offer Choice Page

Смысл: после калькулятора пользователь не сразу видит простую форму, а попадает на страницу выбора сценария.

Сценарии:

- `Пройти подробный опросник` - точнее, дольше, полезнее для ТЗ;
- `Скачать предварительное КП` - быстрее, по данным калькулятора.

Основные требования:

- сохранить выбранную площадь и услуги из калькулятора;
- не доверять client-side totals;
- preliminary КП генерировать только после контакта;
- текущий proposal/PDF flow можно переиспользовать для ветки `Скачать предварительное КП`;
- страница должна объяснять разницу между быстрым КП и подробным опросником.

### PZK-019 - Detailed Questionnaire / Technical Assignment Wizard

Смысл: превратить `Опросный лист.xlsx` в удобный пошаговый wizard.

Основные требования:

- разобрать структуру `docs/design/Опросный лист.xlsx`;
- сгруппировать вопросы по разделам;
- для каждого вопроса определить варианты ответов, `Пока не знаю`, `Пропустить`, `Свой вариант`;
- показывать прогресс;
- сохранять partial progress;
- привязывать ответы к лиду/проекту;
- в админке показывать черновик ТЗ;
- не раскрывать заполненные примерные ответы из XLSX публично, если они содержат клиентские данные.

### PZK-020 - Telegram Delivery And Project Context Bot

Смысл: использовать Telegram не только для внутренних уведомлений, но и как клиентский канал.

Этап 1:

- выдавать пример проекта/КП через Telegram после контакта;
- связать lead с Telegram chat/user;
- логировать факт доставки.

Этап 2:

- после создания проектной Telegram-группы бот может сохранять сообщения/голосовые/файлы;
- раз в сутки агент формирует summary:
  - новые факты для ТЗ;
  - открытые вопросы;
  - противоречия;
  - решения, которые нужно подтвердить;
- в админке появляется черновик обновлений к ТЗ;
- админ вручную принимает или правит изменения.

Важно:

- бот не должен автоматически менять финальное ТЗ без подтверждения;
- нужно явно получить согласие на обработку сообщений/персональных данных;
- Telegram secrets не должны попадать в frontend env или логи;
- Telegram не должен быть единственным каналом, пока не подтверждено, что клиентам это удобно.

## Вопросы для будущей реализации

- Нужен ли strict-gated доступ к PDF examples или достаточно скрыть прямые кнопки за формой?
- Что считать обязательным контактом: телефон, Telegram, или телефон + Telegram?
- Должен ли пример проекта отправляться документом в Telegram или ссылкой?
- Нужно ли уведомлять менеджера о запросе примера отдельно от обычной заявки?
- Будет ли подробный опросник доступен без телефона или только после контакта?
- Можно ли сохранять незавершенный опросник без создания полноценного лида?
- Какой статус в админке получает лид после detailed questionnaire?
- Когда генерировать финальное КП: сразу после questionnaire или после проверки менеджером?

## Рекомендуемый порядок

1. PZK-017 - закрыть утечку примеров без контакта.
2. PZK-018 - изменить сценарий `Получить КП` на страницу выбора.
3. PZK-019 - реализовать подробный questionnaire/TZ wizard.
4. PZK-020 - добавить Telegram delivery, а затем Telegram project context bot.

Такой порядок сохраняет конверсию и не заставляет сразу строить сложного Telegram-агента до того, как в админке появится нормальное место для ТЗ.
