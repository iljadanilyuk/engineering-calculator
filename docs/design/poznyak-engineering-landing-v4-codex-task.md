# PZK-DESIGN-004 - Реализация публичной страницы по визуальному прототипу V4

Дата: 2026-07-17  
Проект: инженерный калькулятор ИП Позняк  
Рабочая папка: `E:\vc\poznyak-engineering-calculator`  
Production: `https://poznyak.by`

## Цель

Реализовать публичную страницу максимально близко к утвержденному визуальному прототипу V4.

Главная задача страницы - продавать услугу проектирования инженерных систем. Калькулятор является центральным интерактивным блоком, но не заменяет смысловую структуру лендинга.

## Файлы-референсы

Положить в `docs/design/`:

- `poznyak-engineering-landing-v4.html` - интерактивный standalone-прототип;
- `poznyak-engineering-landing-v4-reference.png` - утвержденный визуальный референс;
- этот файл - `poznyak-engineering-landing-v4-codex-task.md`;
- исходный скриншот ранее реализованного сайта по проектированию инженерных систем - использовать как референс смыслов и последовательности блоков, но не копировать визуально.

Перед изменением production-кода открыть HTML-прототип в браузере на desktop и mobile.

## Ключевая бизнес-логика

Посетитель должен последовательно понять:

1. Проект нужен до закупки оборудования и начала монтажа.
2. Проект определяет мощности, схемы, трассы, материалы и задания для монтажников.
3. Стоимость зависит от площади, типа объекта и выбранных инженерных разделов.
4. После расчета можно получить коммерческое предложение.
5. Компетентность подтверждается примерами реальной проектной документации и понятным процессом.

Не использовать абстрактные обещания: `лучшее качество`, `надежные решения`, `индивидуальный подход` без конкретного доказательства.

## Структура страницы

### 1. Header

Состав:

- логотип/название `ИП Позняк`;
- подпись `Проектирование инженерных систем`;
- ссылки на калькулятор, пользу проекта, документацию, процесс, FAQ;
- CTA `Получить КП`.

Desktop header - компактный, белый, sticky, с тонкой нижней линией.

Mobile - оставить бренд и CTA. Полное меню можно скрыть или свернуть.

### 2. Hero

Заголовок:

`Проект инженерных систем для дома и небольших объектов`

Подзаголовок:

`Понятные решения и точные расчеты до покупки оборудования и начала монтажных работ.`

Короткие аргументы:

- прозрачный состав работ и материалов;
- схемы и спецификации без лишнего;
- четкие задания для монтажников.

В правой части - реалистичное изображение специалиста и заказчиков, изучающих проектную документацию.

Требования к людям:

- естественная белорусская/восточноевропейская внешность;
- современная, но не постановочная обстановка;
- без американской строительной атрибутики и чужого локального контекста;
- спокойная одежда, частный дом, планы на столе.

Не использовать декоративные псевдочертежи и AI-иллюстрации с бессмысленными линиями.

### 3. Короткий proof/value strip

Четыре пункта:

- Расчеты;
- Схемы;
- Спецификация;
- Задания для монтажа.

Каждый пункт содержит небольшую контурную иконку, заголовок и конкретное пояснение.

### 4. Калькулятор

Калькулятор должен занимать главный визуальный акцент страницы.

Обязательная последовательность элементов:

1. Поле ручного ввода площади.
2. Ползунок площади непосредственно под полем.
3. Переключатель `Частный дом / Коммерческий объект`.
4. Список инженерных разделов.
5. Итоговая стоимость и CTA.

Нельзя переносить выбор типа проекта в hero или выше ввода площади.

#### Ввод площади

- числовое поле;
- единица `м²`;
- диапазон 50-500 м² в прототипе;
- поле и range синхронизированы;
- production-ограничения брать из API/config, если они уже существуют.

#### Тип проекта

Две крупные кнопки:

- `Частный дом`;
- `Коммерческий объект`.

Выбранная кнопка имеет синюю рамку, светло-синюю заливку и понятное активное состояние.

Production-цены для двух типов проекта должны загружаться из backend/API или управляться через админ-панель.

Не переносить демонстрационный коэффициент `1.25` из HTML в production. Он существует только для визуальной демонстрации переключения.

#### Разделы проекта

Состав:

- Проект котельной + 3D;
- Радиаторное отопление;
- Теплые полы;
- Водопровод;
- Канализация;
- Вентиляция;
- Кондиционирование.

Строка содержит:

- название;
- короткое пояснение;
- стоимость;
- стандартный toggle.

Вся строка кликабельна.

Выбранная строка получает спокойную светло-синюю заливку.

Не использовать:

- краны;
- рычаги;
- нестандартные сантехнические переключатели;
- badges с сокращениями КТ, РО, ТП;
- готовые наборы `Отопление / Отопление + ВК / Все системы`.

#### Валюта

На публичной странице показывать только белорусские рубли.

Полностью исключить:

- USD;
- `$`;
- курс валют;
- параллельный пересчет;
- подписи `BYN` в основном интерфейсе.

Использовать новый графический знак белорусского рубля.

В HTML находится CSS-реконструкция для прототипа. В production подключить официальный SVG/asset Национального банка и проверить:

- Chrome;
- Safari;
- Android;
- iOS;
- HTML коммерческого предложения;
- PDF коммерческого предложения.

Оставить текстовый fallback `BYN` только для accessibility/PDF, если официальный SVG не отрисуется.

#### Итоговый блок

Справа на desktop, ниже списка на mobile.

Показывает:

- выбранные разделы;
- стоимость каждого;
- площадь;
- общую стоимость;
- CTA `Получить коммерческое предложение`.

На desktop допустим sticky внутри секции.

### 5. Что дает проектирование

Пять конкретных результатов:

- понимание объема работ и бюджета;
- понимание необходимого оборудования;
- снижение риска переделок;
- понятные задания для монтажников;
- возможность сравнивать поставщиков и предложения на единой основе.

Не использовать одинаковые большие карточки. Сохранить легкую редакционную сетку и небольшие контурные иконки.

### 6. Примеры проектной документации

Показать четыре типа результата:

- планы и схемы;
- 3D-визуализация;
- спецификации;
- узлы и детали.

Использовать реальные превью страниц из:

- `website/public/project-examples/proekt_primer_ov.pdf`;
- `website/public/project-examples/primer_proekt_vk.pdf`.

Не использовать нарисованные кодом схемы или случайные AI-чертежи.

Для production подготовить четкие JPG/WebP-превью реальных листов. Карточки должны вести к соответствующим PDF.

### 7. Как проходит работа

Пять этапов:

1. Заявка.
2. Сбор данных.
3. Расчеты и проект.
4. Согласование.
5. Выдача проекта.

Desktop - горизонтальная линия этапов.

Mobile - вертикальная или двухколоночная структура без горизонтального скролла.

### 8. FAQ

Вопросы:

- зачем нужен проект до покупки оборудования;
- что входит в проект;
- сроки разработки;
- возможность корректировок;
- формат передачи;
- работа по всей Беларуси.

Использовать доступный accordion:

- `button`;
- `aria-expanded`;
- управление клавиатурой;
- плавное, но короткое раскрытие.

### 9. Финальный CTA

Темный navy-блок.

Состав:

- заголовок;
- краткое объяснение следующего шага;
- форма;
- контакты;
- факты о работе по договору.

Не публиковать вымышленные телефоны и email. Брать контакты из фактического config/env/CMS. Если данных нет, оставить задачу заблокированной до подтверждения владельца.

## Визуальная система

### Палитра

- основной navy: `#081A2F`;
- дополнительный navy: `#0D2B4C`;
- синий акцент: `#0B5FB5`;
- темный текст: `#102033`;
- основной фон: `#FFFFFF`;
- светлый фон секции: `#F6F8FB`;
- линии: `#DBE3EB`;
- активная заливка: `#F2F7FD`.

Не использовать бежево-оранжевую палитру.

Не использовать фиолетовые градиенты, glow, glassmorphism или неон.

### Типографика

Предпочтительно:

- `Inter`, `Manrope` или другой нейтральный современный sans-serif с хорошей кириллицей;
- крупный, но не рекламно-гипертрофированный H1;
- высокая читаемость интерфейса калькулятора;
- tabular numerals для цен.

Не использовать декоративные шрифты и чрезмерно отрицательный tracking.

### Радиусы и тени

- умеренные радиусы 8-16 px;
- карточки не должны выглядеть как набор одинаковых SaaS-компонентов;
- тени очень мягкие и только для иерархии;
- не оборачивать каждый текстовый блок в карточку.

## Responsive

Проверить минимум:

- 1440 px;
- 1280 px;
- 1024 px;
- 768 px;
- 390 px.

Mobile-порядок калькулятора:

1. площадь;
2. range;
3. тип проекта;
4. разделы;
5. итог;
6. CTA.

На 390 px:

- нет горизонтального скролла;
- зоны переключателей не меньше 44 px;
- итог читается без увеличения;
- подписи разделов не обрезаются;
- sticky-итог не перекрывает интерфейс;
- форма удобна для телефона.

## Технические ограничения

Не менять без отдельной задачи:

- backend-side recalculation;
- calculation domain;
- public lead submit flow;
- immutable calculation snapshot;
- proposal HTML/PDF generation;
- token-protected proposal links;
- admin routes;
- DB schema;
- DigitalOcean;
- DNS;
- env/secrets.

Фронтенд не является источником истины по цене.

После отправки заявки backend обязан повторно пересчитать сумму по актуальным тарифам.

## Data mapping

Из API/config брать:

- допустимый диапазон площади;
- типы проекта;
- список услуг;
- описание услуг;
- тип тарифа: фиксированный или за м²;
- активность услуги;
- порядок вывода;
- публичную цену в белорусских рублях;
- итог;
- контакты;
- ссылки на реальные PDF.

Статически можно оставить:

- тексты hero;
- смысловые блоки;
- этапы;
- FAQ;
- подписи примеров документации.

## Success state

После успешной отправки заявки сохранить существующий сценарий:

- подтверждение;
- ссылка на HTML КП;
- ссылка на PDF КП;
- корректная обработка ошибок;
- повторная отправка не должна создавать случайные дубли.

Визуально success state должен продолжать стиль страницы, а не выглядеть как системный alert.

## Accessibility

Обязательно:

- семантические `header`, `main`, `section`, `form`;
- label для каждого input;
- переключатели услуг как native checkbox или `button role="switch"`;
- `aria-checked`;
- видимый `focus-visible`;
- достаточный контраст;
- alt для изображений;
- клавиатурное управление modal и FAQ;
- закрытие modal по Escape;
- блокировка фокуса внутри modal, если используется dialog.

## Порядок реализации

1. Создать отдельную feature branch.
2. Изучить текущий `website/src/pages/index.astro`.
3. Зафиксировать существующие API-вызовы и states.
4. Разбить страницу на компоненты, не меняя backend contracts.
5. Реализовать визуал по HTML-прототипу.
6. Подключить реальные данные API.
7. Подключить реальные PDF-превью.
8. Проверить desktop/mobile.
9. Проверить lead submit, success state, HTML КП и PDF КП.
10. Не выполнять production deployment до визуального согласования.

## Критерии приемки

- первый экран выглядит максимально близко к референсу V4;
- изображения соответствуют белорусскому/восточноевропейскому контексту;
- переключатель типа проекта находится после ввода площади и range;
- используются обычные toggles;
- публично отображаются только белорусские рубли;
- нет `$`, USD и курсов;
- калькулятор работает с backend-side recalculation;
- используются реальные примеры проектной документации;
- страница не выглядит как бежево-оранжевый AI-шаблон;
- нет горизонтального overflow на 390 px;
- текущие production-сценарии не сломаны.
