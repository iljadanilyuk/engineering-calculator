# PZK-ADMIN-DESIGN-002 - административная панель V2

Дата: 2026-07-17
Проект: `poznyak-engineering-calculator`
Рабочая папка: `E:\vc\poznyak-engineering-calculator`
Текущая админка: `https://admin.poznyak.by`
Интерактивный прототип: `poznyak-admin-prototype-v2.html`

## 1. Цель

Пересобрать административную панель в единое рабочее пространство для полного цикла инженерного проекта:

`Заявка -> КП -> ТЗ -> Договор -> Проектирование -> Проверка и выдача -> Завершено`

Главное изменение относительно V1: подписание договора не считается завершением проекта. Договор запускает проектирование. Завершение разрешено только после проверки документации, сверки с ТЗ и подтвержденными договоренностями, получения финальной оплаты и выдачи спецификаций.

Production-код, базу данных, DNS, DigitalOcean, env и действующие API не менять без отдельного подтверждения.

## 2. Что подтверждено и обязательно сохраняется

- постоянный левый sidebar;
- канбан и таблица проектов;
- единая карточка проекта;
- обязательное следующее действие с датой, временем и ответственным;
- отдельный раздел задач;
- versioned questionnaire;
- отдельный конструктор опросника;
- отдельные тарифы private/commercial;
- функциональные цветовые статусы;
- Telegram-сообщение становится договоренностью только после подтверждения менеджером;
- договор создается из согласованного КП и проверенного ТЗ;
- calculation, proposal, questionnaire и contract snapshots не изменяются задним числом;
- основной desktop-сценарий: 1280-1600 px;
- production deployment запрещен до отдельного подтверждения.

## 3. Ключевые бизнес-правила V2

### 3.1. Договор запускает работу

Статус `Подписан` означает:

- договор зафиксирован snapshot;
- проект можно передать проектировщику;
- создается стартовая задача;
- назначаются ответственный проектировщик и плановый срок;
- проект переходит в макроэтап `Проектирование`.

Статус подписанного договора не может автоматически закрывать проект.

### 3.2. Завершение проекта требует проверки результата

Перед закрытием проекта менеджер проверяет:

1. Все ли разделы из согласованного КП представлены в комплекте документации.
2. Все ли обязательные требования ТЗ отражены в проекте.
3. Все ли подтвержденные договоренности с клиентом учтены.
4. Есть ли для каждого требования ссылка на лист, узел, файл или примечание проекта.
5. Закрыты ли замечания проектировщика, менеджера и клиента.
6. Подготовлен ли комплект проекта без спецификаций.
7. Отправлен ли этот комплект клиенту.
8. Получена ли вторая часть оплаты.
9. Отправлены ли спецификации после оплаты.

Проект нельзя перевести в `Завершено`, пока обязательные пункты не выполнены.

### 3.3. Матрица требований

Нужна трассируемость требований:

`Источник -> Требование -> Подтверждение -> Где отражено в проекте -> Статус проверки`

Источники:

- ответ в ТЗ;
- подтвержденная договоренность из Telegram;
- подтвержденная договоренность из звонка или встречи;
- комментарий менеджера, оформленный как требование;
- приложение к договору.

Статусы требования:

- `Не проверено`;
- `Подтверждено в проекте`;
- `Требует уточнения`;
- `Не отражено`;
- `Отклонено с причиной`;
- `Не применимо`.

Для подтверждения менеджер должен указать evidence:

- документ;
- номер листа;
- номер узла или примечания;
- комментарий;
- при необходимости файл или изображение.

### 3.4. Выдача документации в два этапа

#### Пакет 1 - проект без спецификаций

Можно отправить после внутренней проверки документации.

В пакет входят:

- чертежи;
- схемы;
- пояснения;
- согласованные проектные решения;
- другие документы, разрешенные до финальной оплаты.

В пакет не входят спецификации оборудования и материалов.

#### Пакет 2 - спецификации

Отправка разрешена только после подтверждения второй части оплаты.

В пакет входят:

- спецификации оборудования;
- спецификации материалов;
- связанные XLSX/PDF;
- другие документы с правилом доступа `После финальной оплаты`.

До оплаты кнопка отправки заблокирована. Ручная разблокировка допускается только администратору, требует причины и записывается в audit log.

### 3.5. Следующее действие

У каждого активного проекта обязательно есть:

- тип действия;
- текст действия;
- дата и время;
- ответственный;
- напоминание;
- связь с проектом;
- статус `Просрочено / Сегодня / Запланировано / Выполнено`.

После выполнения система предлагает сразу создать следующее действие.

## 5. Модель этапов

### 5.1. Макроэтапы активной воронки

1. `new` - Новая заявка.
2. `proposal` - КП.
3. `questionnaire` - ТЗ.
4. `contract` - Договор.
5. `design` - Проектирование.
6. `handover` - Проверка и выдача.

Терминальные состояния:

- `completed` - Завершено;
- `rejected` - Отказ;
- `spam` - Спам;
- `archived` - Архив.

### 5.2. Статусы документов хранятся отдельно

Примеры:

- КП: черновик, отправлено, открыто, требует корректировки, согласовано, отклонено;
- ТЗ: не отправлено, заполняется, остановилось, требует проверки, требует уточнения, проверено;
- договор: черновик, требует проверки, отправлен, ожидает подписи, подписан, отменен;
- проектирование: не начато, в работе, внутренняя проверка, корректировка, готово;
- сдача: проверка требований, пакет без спецификаций отправлен, ожидается оплата, спецификации разблокированы, спецификации отправлены.

Макроэтап, статус документа и срочность следующего действия не объединять в одно поле.

Два пакета:

Исходное сообщение неизменяемо. Подтвержденная договоренность хранится отдельной сущностью со ссылкой на исходное событие.

- кто создал;
- кто отправил;

Правила выдачи:

- внутренний;
- можно отправить до финальной оплаты;

- отправлен клиенту;

## 11. Конструктор опросника

Трехколоночный интерфейс:

1. Разделы.
2. Вопросы выбранного раздела.
3. Настройки вопроса.

- архив.

Типы вопросов:

- короткий текст;

- число;
- дата;
- один вариант;

- да / нет / не знаю;

- повторяемая группа;
- информационный блок.

Условная логика может зависеть от:

- выбранной услуги в согласованном КП;

- типа объекта;

## 13. Договоры

Колонки:

- договор;
- клиент и проект;

- статус договора;

## 14. Интеграции

Telegram:

1. Бот получает сообщения только из привязанной группы.
2. Исходное сообщение сохраняется неизменяемым.
3. Система может предложить сводку.
4. Менеджер подтверждает договоренность.
5. Подтвержденная договоренность становится требованием проекта.
6. Автоматическое создание задач не включать до отдельного тестирования.

## 15. Роли и доступ

- Администратор - полный доступ;
- Менеджер - клиенты, задачи, КП, договоры, проверка требований, оплата и выдача;

Только администратор может вручную обойти блокировку спецификаций. Действие обязательно пишется в audit log.

## 17. State machine сдачи

Рекомендуемая последовательность:

1. `design_in_progress`.
2. `engineering_review`.
3. `manager_requirement_review`.
4. `needs_correction` или `ready_for_preview_delivery`.
5. `project_without_specs_sent`.
6. `awaiting_final_payment`.
7. `final_payment_received`.
8. `specifications_released`.
9. `completed`.

### Обязательные gates

Для `ready_for_preview_delivery`:

- все договорные разделы готовы;
- инженерная проверка завершена;
- нет критических ошибок комплекта.

Для `final_payment_received`:

- платеж подтвержден уполномоченным пользователем;
- сохранена дата и источник подтверждения.

Для `specifications_released`:

- финальная оплата подтверждена;
- существует актуальная версия спецификаций;
- отправка записана в историю.

Для `completed`:

- все обязательные требования имеют финальный статус;
- обязательный чек-лист выполнен;
- пакет без спецификаций отправлен;
- финальная оплата получена;
- спецификации отправлены;
- отсутствуют открытые критические замечания.

## 20. Responsive

Проверить:

- 1440x900;
- 1280x800;
- 1024x768;
- 390 px для просмотра карточки и задач.

Правила:

- канбан допускает горизонтальный scroll;
- карточка проекта перестраивается в одну колонку;
- sidebar на mobile открывается кнопкой;
- таблицы получают горизонтальный scroll;
- inspector конструктора на 1024 px открывается в drawer;
- редактирование конструктора и тарифов на mobile не является обязательным сценарием;
- основные задачи, коммуникация и документы должны оставаться доступными.

- модели;
- routes;
- serializers/schemas;
- admin API;
- миграции;
- генерацию КП и PDF;
- хранение документов;
- платежные данные;

## 23. Критерии приемки V2

- постоянный sidebar;
- отдельный экран задач;
- рабочий стол показывает действия и блокеры, а не повторяющиеся KPI;
- канбан использует шесть активных макроэтапов;
- таблица проектов доступна;
- макроэтап отделен от статуса документа;
- следующее действие видно без прокрутки;
- карточка проекта связывает КП, ТЗ, договор, проектирование, коммуникацию и документы;
- подписание договора запускает проектирование, но не завершает проект;
- есть матрица требований с evidence;
- есть чек-лист завершения;
- проект без спецификаций можно отправить отдельно;
- спецификации заблокированы до финальной оплаты;
- подтверждение оплаты разблокирует отправку спецификаций;
- ручной обход блокировки записывается в audit log;
- опросник содержит 9 разделов и 89 вопросов;
- опубликованная версия опросника неизменяема;
- тарифы private/commercial раздельные;
- таблица услуг по умолчанию не находится в режиме редактирования;
- используется Inter 400/500/600/700;
- числовые колонки используют tabular figures;
- интерфейс читаем на 1440, 1280 и 1024 px;
- production не изменен.

## 24. Сценарии ручного тестирования

### Сценарий A - новая заявка

1. Открыть проект из канбана.
2. Проверить расчетный snapshot.
3. Создать следующее действие.
4. Подготовить КП.

### Сценарий B - согласование КП и ТЗ

1. Создать новую версию КП.
2. Отправить клиенту.
3. Зафиксировать согласование.
4. Создать ТЗ из опубликованной версии.
5. Проверить, что последующие изменения шаблона не меняют snapshot.

### Сценарий C - договор

1. Попытаться создать договор при непроверенном ТЗ.
2. Получить блокировку с понятной причиной.
3. Проверить ТЗ.
4. Создать договор.
5. Подписать договор.
6. Убедиться, что проект перешел в проектирование, а не в завершено.

### Сценарий D - финальная проверка

1. Открыть этап `Проверка и выдача`.
2. Найти требование без evidence.
3. Привязать лист проекта.
4. Закрыть обязательные пункты чек-листа.
5. Отправить проект без спецификаций.

### Сценарий E - финальная оплата и спецификации

1. Попытаться отправить спецификации до оплаты.
2. Получить блокировку.
3. Подтвердить финальную оплату.
4. Проверить дату, пользователя и audit event.
5. Отправить спецификации.
6. Закрыть проект после выполнения всех обязательных условий.
